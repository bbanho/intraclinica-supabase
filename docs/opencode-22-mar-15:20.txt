Bug real encontrado: /clinical redireciona para /inventory
- ClinicalComponent importa LocalAiService, que faz import '@tensorflow/tfjs-backend-webgpu' no nível de módulo
- O TensorFlow/WebGPU falha ao inicializar em ambientes sem GPU (e em headless browser)
- O erro: TypeError: Class extends value undefined is not a constructor or null em @tensorflow_tfjs.js
- O Angular lazy-loader captura a exceção e o wildcard route { path: '**', redirectTo: '' } redireciona para inventory
- Consequência: a rota /clinical está inacessível em qualquer browser sem WebGPU — não é só problema de teste headless
- Solução necessária: o import do TF/WebGPU deve ser lazy/dinâmico dentro do método checkWebGPUSupport(), não no topo do arquivo

🔴 Bug real a corrigir (próxima tarefa)
local-ai.service.ts — import TensorFlow no topo do arquivo quebra lazy-loading do ClinicalComponent
Arquivo: /var/home/bruno/Documentos/intraclinica-supabase/frontend/src/app/core/services/local-ai.service.ts
Linhas problemáticas:
import * as tf from '@tensorflow/tfjs';          // linha 2
import '@tensorflow/tfjs-backend-webgpu';         // linha 3
Esses imports executam código de inicialização do TF/WebGPU no momento do carregamento do módulo. Em qualquer ambiente sem WebGPU (incluindo browsers comuns sem GPU), isso lança TypeError: Class extends value undefined e faz o Angular redirecionar para o wildcard route.
Solução: converter para import dinâmico dentro do método checkWebGPUSupport():
async checkWebGPUSupport() {
  try {
    const tf = await import('@tensorflow/tfjs');
    await import('@tensorflow/tfjs-backend-webgpu');
    // ... resto da lógica
  } catch (e) {
    this.hasWebGPU.set(false);
  }
}
