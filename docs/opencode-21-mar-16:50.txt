Discoveries
Problema central: God Service + 3 padrões de state
DatabaseService (759L) faz tudo: auth, state de 10 entidades, mappers, RPCs, permission check, context switch. Três padrões de state coexistem sem dono claro: DatabaseService signals, NgRx, e direct service calls.
Dois InventoryService com responsabilidades sobrepostas
- core/services/inventory.service.ts — usado por InventoryComponent, ClinicalExecutionComponent, ProcedureRecipeComponent. Retorna tipo InventoryItem.
- features/inventory/data/inventory.service.ts — usado por InventoryEffects (NgRx). Retorna tipo Product. Tem getProducts, getTransactions, addProduct, deleteProduct.
O InventoryEffects precisa ser reescrito para usar core/services/inventory.service.ts. O mapeamento de tipos precisa ser reconciliado: Product (em types.ts) usa stock/minStock/costPrice/batchNumber (camelCase), enquanto InventoryItem (em inventory.types.ts) usa campos ligeiramente diferentes. Verificar o que InventoryStore facade e ReportsComponent esperam antes de alterar os tipos.
Sentinel 'all' — convenção crítica
selectedContextClinic usa null (não inicializado), 'all' (SUPER_ADMIN global — não é UUID válido), ou UUID. Todo código que passa esse valor para Supabase DEVE guardar !== 'all'.
Rotas problemáticas identificadas
- /clinical-execution → duplicata de /clinical (mesmo componente)
- /procedures, /pacientes, /consultas → sem link no sidebar, inacessíveis pela UI
IAM frontend-only
db.checkPermission() roda no browser. Sem RLS correspondente verificado — risco de segurança não corrigido neste plano (anotado para depois).
State machine de agendamento frontend-only
DatabaseService.transitionAppointmentStatus() existe mas ReceptionComponent usa store.updateAppointmentStatus() que bypassa a validação — inconsistência a resolver na Fase 2.
Fase 4: Schema das novas tabelas (já especificado em REFACTORING_PLAN.md)
ui_module (id, key unique, label, icon, route, description, created_at)
clinic_module (id, clinic_id FK→clinic, module_key FK→ui_module.key, enabled, sort_order)
clinic_config (id, clinic_id FK→clinic, key, value jsonb)
RPC get_clinic_ui_config retorna { modules: UiModule[], config: Record<string, unknown> }.
