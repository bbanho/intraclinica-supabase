Bug crítico detectado durante smoke-test: Login race condition
O smoke-test com Playwright revelou um bug real no fluxo de login:
Sintoma: Login Supabase Auth retorna HTTP 200 (autenticação bem-sucedida), dados são carregados (product, appointment, etc. — todos HTTP 200/300), mas o Angular permanece em /#/login sem navegar.
Root cause identificada:
1. NgRx login$ effect → signInWithPassword → sucesso → dispara loadSession()
2. loadSession$ effect → getUser() → monta profile com user_metadata['name'] e app_metadata['role'] → dispara loginSuccess
3. loginSuccess$ effect → router.navigate(['/'])
4. authGuard verifica db.currentUser() (signal do DatabaseService)
5. RACE CONDITION: DatabaseService.currentUser é preenchido via supabase.auth.onAuthStateChange (assíncrono), mas o guard checa o signal ANTES do onAuthStateChange ter executado e preenchido o signal
6. Guard encontra currentUser() === null → redireciona de volta para /login
Arquivos relevantes para o fix:
- frontend/src/app/core/guards/auth.guard.ts — verifica db.currentUser() (síncrono)
- frontend/src/app/core/store/auth/auth.effects.ts — loginSuccess$ navega para / imediatamente
- frontend/src/app/core/services/database.service.ts — currentUser preenchido em onAuthStateChange (linha ~104-143)
- frontend/src/app/features/login/login.component.ts — tem "Auto-Login Hack" que checa localStorage token no ngOnInit e navega para /inventory se token presente
O Auto-Login Hack no login.component.ts (linha ~307):
const token = localStorage.getItem('sb-prolahgqlwfriwfpzjdm-auth-token');
if (token) {
    this.router.navigate(['/inventory']);
}
Isso funciona na segunda visita (token já em localStorage), mas não no primeiro login.
Fix proposto: O authGuard deve ser assíncrono — esperar o onAuthStateChange inicial antes de tomar decisão. Ou o guard deve verificar a sessão Supabase diretamente em vez de depender do signal já populado.
Arquitetura NgRx vs DatabaseService (dois sistemas paralelos)
O app tem dois sistemas de estado paralelos que não estão bem sincronizados:
1. NgRx Store — auth.effects.ts com login$, loadSession$, loginSuccess$
2. DatabaseService signals — currentUser, users, appointments, etc. preenchidos via onAuthStateChange
O guard usa o sistema 2, o NgRx usa o sistema 1 para navegar — daí a race condition.
